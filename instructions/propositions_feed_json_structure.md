# Propositions Feed JSON Structure

This document describes the JSON structure currently generated by:

- `feed_module.propositions.generate_propositions_xml(...)`

The function name still contains `xml`, but the current implementation writes JSON.

## Purpose

This feed represents proposition-level commerce data:

- price
- old price
- availability
- warranty
- installments
- dispatch timing
- delivery methods
- optional checkout and manufacture metadata

Current public endpoint:

- `GET /api/propositions-feed.xml`

Important:

- the URL still ends with `.xml`
- the response body is JSON

## Relation To `RESULTING_FEED_STRUCTURE_EXAMPLE.json`

The file:

- `instructions/RESULTING_FEED_STRUCTURE_EXAMPLE.json`

is clearly a propositions-style structure reference.

The current generator follows the same logical top-level shape:

```json
{
  "total": 0,
  "data": []
}
```

Differences:

- the generator writes valid JSON
- the generator always includes a stable set of keys in each proposition object
- default values follow the current implementation and the PDF/MD rules, not every value shown in that example file

## Top-Level JSON Shape

```json
{
  "total": 18,
  "data": [
    {
      "code": "10337554",
      "price": 680,
      "old_price": null,
      "availability": true,
      "warranty_type": "no",
      "warranty_period": 0,
      "max_pay_in_parts": 3,
      "delivery_methods": [
        {
          "method": "nova-post:branch",
          "price": 0
        }
      ],
      "days_to_dispatch": 1,
      "checkout_constraints": null,
      "manufacture": null
    }
  ]
}
```

Top-level keys:

- `total`: integer count of exported propositions
- `data`: array of proposition objects

`total` is always equal to `len(data)` in the current implementation.

## Inclusion Rules

An item is exported when a barcode can be resolved.

Current barcode resolution order:

1. source offer `<barcode>`
2. supplemental source offer `<barcode>` matched by the same `offer id`

Important:

- unlike the content feed, propositions are not filtered by `availability`
- unavailable items may still be exported with `"availability": false`
- barcode is used only as an export filter
- barcode is not included as a field in the propositions JSON object

## Proposition Object Structure

Each item in `data[]` is a JSON object with these keys, in this current order:

1. `code`
2. `price`
3. `old_price`
4. `availability`
5. `warranty_type`
6. `warranty_period`
7. `max_pay_in_parts`
8. `delivery_methods`
9. `days_to_dispatch`
10. `checkout_constraints`
11. `manufacture`

All of these keys are always present in every exported proposition.

## Field Reference

| Key | Type | Nullable / Empty | Source / Logic | Notes |
|---|---|---:|---|---|
| `code` | `string` | No in normal flow | `override.code` or `source_offer.offer_id` | Main identifier for proposition matching/order flow. |
| `price` | `int \| null` | Yes | `override.price` or source `price` | Required by business rules, but current JSON can still contain `null` when source data is missing and `strict=False`. |
| `old_price` | `int \| null` | Yes | `override.old_price` or source old price | Present on every item, `null` when absent. |
| `availability` | `boolean` | No | `override.availability` or `source_offer.available` | Real boolean, not string. |
| `warranty_type` | `string` | No | `override.warranty_type` | Default is `"no"`. Allowed business values are `manufacturer`, `merchant`, `no`. |
| `warranty_period` | `int` | No | `override.warranty_period` | Default is `0`. |
| `max_pay_in_parts` | `int` | No | `override.max_pay_in_parts` | Default is `3`. |
| `delivery_methods` | `array[object]` | Yes, can be `[]` | `override.delivery_methods` | Default is the full built-in list of delivery methods. |
| `days_to_dispatch` | `int` | No | `override.days_to_dispatch` | Default is `1`. |
| `checkout_constraints` | `object \| null` | Yes | Built from `override.multiplicity` | `null` when absent. |
| `manufacture` | `object \| null` | Yes | Built from `override.country_code` and `override.manufacture_year` | `null` when both are absent. |

## `delivery_methods` Field

`delivery_methods` is an array of objects with this shape:

```json
{
  "method": "nova-post:branch",
  "price": 0
}
```

Current default methods come from `DEFAULT_DELIVERY_METHODS`:

1. `nova-post:branch`
2. `nova-post:cargo_branch`
3. `nova-post:postomat`
4. `courier:nova-post`

Default price for every method is `0`.

If an override supplies a custom `delivery_methods` list, that list is used as-is.

## `checkout_constraints` Field

When `override.multiplicity` is missing:

```json
"checkout_constraints": null
```

When present:

```json
"checkout_constraints": {
  "multiplicity": 2
}
```

## `manufacture` Field

When both manufacture fields are missing:

```json
"manufacture": null
```

When present:

```json
"manufacture": {
  "country_code": "CZ",
  "year": 2025
}
```

Important:

- if only one of the two manufacture values is provided, the object is still emitted
- the missing inner field remains `null`

Example:

```json
"manufacture": {
  "country_code": "CZ",
  "year": null
}
```

## Implementation Notes

- The current function writes JSON to disk even though its name is `generate_propositions_xml`.
- The JSON structure matches the logical shape described in the propositions PDF much more closely than the previous XML wrapper.
- If `strict=True`, the function validates required business fields after building the payload and raises an error when `code` or `price` is missing.
- The structure remains stable because optional blocks are represented by `null` instead of omission.
